# 05 — Agent Architecture

> **In plain English:** MiroFlow uses a "boss and worker" pattern. A **main agent** thinks about the big picture and delegates specific tasks to **sub-agents** who do the hands-on work (searching, reading files, etc.). This page explains how that hierarchy works and how the prompt system generates instructions for each agent.

---

## Main Agent / Sub-Agent Hierarchy

```
┌─────────────────────────────────────────────────────┐
│                   Main Agent                         │
│  - Has access to reasoning tools                     │
│  - Orchestrates high-level task strategy             │
│  - Can delegate subtasks to sub-agents               │
│  - Generates final summary with \boxed{} answer     │
│                                                      │
│  Tool access: reasoning (+ sub-agent delegation)     │
└──────────┬──────────────────────────────────────────┘
           │  Delegates via tool calls to "agent-*"
           ▼
┌─────────────────────────────────────────────────────┐
│              Sub-Agent (agent-worker)                 │
│  - Executes specific information-gathering subtasks  │
│  - Has access to web search, file reading, code,     │
│    vision, audio tools                               │
│  - Returns detailed findings to main agent           │
│                                                      │
│  Tool access: search, read, code, vision, audio      │
└─────────────────────────────────────────────────────┘
```

### How It Works

1. **The main agent** receives the user's question.
2. It thinks about what information it needs and formulates a plan.
3. Instead of searching the web itself, it **calls `agent-worker` as if it were a tool**, passing a specific subtask like "Search for recent GDP data for France."
4. The sub-agent runs its *own* loop — searching, reading, analyzing — until it has an answer.
5. The sub-agent's answer is returned to the main agent as a "tool result."
6. The main agent integrates all information and produces the final answer.

### The Key Trick: Sub-Agents *Are* Tools

From the main agent's perspective, calling `agent-worker` is no different from calling `google_search` or `read_file`. It's just another tool. The orchestrator detects that the tool name starts with `"agent-"` and transparently spawns a full sub-agent loop instead of calling an MCP server.

This is implemented in:
- **`tool_utils.py::expose_sub_agents_as_tools()`** — Creates tool definitions from sub-agent prompt classes.
- **`orchestrator.py`** — When `server_name.startswith("agent-")`, triggers `run_sub_agent()` instead of MCP tool execution.

> **Design rationale — Sub-agents as tools:**
> 1. **Simplicity:** The main agent doesn't need special logic for delegation — it just "calls a tool."
> 2. **Composability:** You can add more sub-agents (e.g., a "code specialist" or "math specialist") by just adding them to the config.
> 3. **Recursive potential:** In theory, a sub-agent could itself delegate to another sub-agent, enabling arbitrarily deep agent hierarchies.
> 4. **Separation of concerns:** The main agent focuses on *what* to do; the sub-agent focuses on *how* to do it.

## The Prompt System

Prompts (the instructions given to each agent) are generated by Python classes, not hardcoded strings. This makes them dynamic and customizable.

### Class Hierarchy

```
BaseAgentPrompt (ABC)           ← Abstract base class
├── MainAgentPrompt_GAIA        ← Prompt for GAIA benchmark
├── MainAgentPromptBoxedAnswer  ← General-purpose boxed-answer prompt
├── MainAgentPrompt_DeepSeek    ← DeepSeek-specific prompt
└── SubAgentWorkerPrompt        ← Prompt for sub-agent workers
```

### What Each Prompt Class Does

| Method | Purpose |
|--------|---------|
| `generate_system_prompt_with_mcp_tools(…)` | Creates the full system prompt, including descriptions of all available tools in XML format |
| `generate_summarize_prompt(…)` | Creates the "wrap up and give your final answer" prompt |
| `expose_agent_as_tool(…)` | (Sub-agents only) Creates a tool definition so the main agent can call this agent |

### How Tool Calls Work in Prompts

MiroFlow uses **XML-style tool calls** for most providers:

```xml
<use_mcp_tool>
  <server_name>tool-searching</server_name>
  <tool_name>google_search</tool_name>
  <arguments>
  {"query": "future event prediction 2025"}
  </arguments>
</use_mcp_tool>
```

For OpenAI models (GPT-4o, GPT-5, etc.), MiroFlow uses OpenAI's **native function-calling API** instead. The system automatically converts MCP tool definitions to the OpenAI format.

> **Design rationale — XML tool calling for Anthropic:**
> Using XML-formatted tool calls (parsed with regex) gives MiroFlow full control over the tool-calling format. It works uniformly across different providers and avoids vendor lock-in to any specific function-calling API. For OpenAI models, native function calling is used because it's more reliable with those models.

> **Design rationale — Prompt classes instead of string templates:**
> Python classes allow prompts to include dynamic logic — like listing available tools, adjusting for language (Chinese vs. English), or changing behavior based on the benchmark. This is much more powerful than static template strings and enables easy A/B testing of prompt strategies.
